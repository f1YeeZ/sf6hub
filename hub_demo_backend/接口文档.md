# 后端接口对接说明

更新日期：2026-07-11

本文档对应当前 Spring Boot Controller 与前端 `src/js/api/index.js`。所有路径均以 `/api` 为基础路径；示例开发地址为 `http://localhost:8080/api`。

## 通用约定

### 认证 Cookie

- 普通用户：`hub_auth`，HttpOnly，`SameSite=Lax`，路径 `/api`。
- 后台用户：`hub_admin_auth`，HttpOnly，`SameSite=Lax`，路径 `/api`。
- 登录接口不向 JavaScript 返回 JWT，响应中的 `token` 字段为空字符串；浏览器通过 Cookie 保持会话。
- JWT 有效期为 7 天，并包含用户 `token_version`。修改或重置密码后，旧普通/后台会话立即失效。
- Cookie 是否带 `Secure` 由 `PUBLIC_BASE_URL` 是否为 HTTPS 决定；生产必须使用 HTTPS。

### CSRF

后端采用双提交 Cookie：

1. 先请求 `GET /health` 获取 `hub_csrf` Cookie 和 `X-CSRF-Token` 响应头。
2. 除 GET、HEAD、OPTIONS 外的所有方法（包括未来可能增加的 PATCH）必须同时携带 Cookie 和同值的 `X-CSRF-Token` 请求头。
3. `hub_csrf` 可被 JavaScript 读取，`SameSite=Lax`，路径 `/api`，最长 7 天；登录时轮换，普通用户退出时清除。
4. 受反爬保护的 GET 也需要有效 CSRF：角色连招分页/筛选选项、连招详情/后续路线、用户连招/收藏列表。
5. 前端 API 客户端会优先复用内存 token 或 Cookie；缺失时请求 `/health`。收到 CSRF 403 后当前不会自动重试，调用方应重新初始化 token 后再发起操作。

前端请求使用 `credentials: include`。正式支持并推荐前后端同源的 Nginx `/api` 代理部署；跨源部署除了精确 CORS 外还受浏览器 SameSite Cookie 规则限制，不应在未做浏览器兼容验证时作为生产拓扑。

### 分页

分页响应：

```json
{
  "list": [],
  "page": 1,
  "pageSize": 12,
  "total": 0
}
```

普通接口最大 `pageSize=50`，后台接口最大 `pageSize=100`。

### 错误响应

业务错误、参数错误和限流错误使用统一结构：

```json
{
  "success": false,
  "message": "错误说明",
  "data": null
}
```

常见状态码：400 参数/业务错误、401 未登录、403 CSRF 或权限不足、429 请求过多、500 未处理异常。

## 健康与实时事件

- `GET /health`：应用和数据库健康状态，同时初始化 CSRF token。
- `GET /events`：前台 SSE，事件名 `realtime-ready`、`realtime-update`。
- `GET /admin/events`：后台 SSE，需要管理员或 `combo_review` 权限。

## 用户认证 `/auth`

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/auth/register` | 使用用户名、邮箱和密码注册，并设置普通认证 Cookie。 |
| POST | `/auth/login` | 使用邮箱和密码登录。 |
| POST | `/auth/admin/login` | 管理员或连招审核员后台登录。 |
| POST | `/auth/refresh` | 验证当前普通会话并轮换 Cookie。 |
| POST | `/auth/logout` | 清除普通认证和 CSRF Cookie。 |
| POST | `/auth/admin/logout` | 清除后台认证 Cookie。 |
| POST | `/auth/password-reset/send-code` | 发送密码重置验证码。 |
| POST | `/auth/password-reset/verify-code` | 验证邮箱验证码并返回一次性 `resetToken`。 |
| POST | `/auth/password-reset/reset` | 使用 `resetToken` 修改密码并撤销旧会话。 |

注册请求：

```json
{
  "username": "player_1",
  "email": "player@example.com",
  "password": "new-secret-123"
}
```

登录请求：

```json
{
  "email": "player@example.com",
  "password": "new-secret-123"
}
```

登录/注册响应示例：

```json
{
  "id": 1,
  "username": "player_1",
  "email": "player@example.com",
  "token": "",
  "role": "user",
  "avatar": null,
  "adminPermissions": ""
}
```

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/me` | 当前普通用户私有资料。 |
| PUT | `/profile/username` | 修改用户名。 |
| POST | `/profile/email/current-code` | 向当前邮箱发送验证代码。 |
| POST | `/profile/email/new-code` | 向新邮箱发送验证代码。 |
| PUT | `/profile/email` | 使用当前/新邮箱双验证码修改邮箱。 |
| POST | `/profile/password/code` | 发送密码修改验证码。 |
| PUT | `/profile/password` | 修改密码并撤销旧会话。 |

| GET | `/users/{id}/favorite-combos` | 用户收藏的已审核连招分页。 |

资料更新采用字段级 SQL，不会把并发修改的其他资料字段写回旧值。同一字段的并发修改采用“最后提交生效”。

修改邮箱请求：

```json
{
  "newEmail": "new@example.com",
  "currentEmailCode": "111111",
  "newEmailCode": "222222"
}
```

## 角色、帧数据与连招

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/characters` | 角色列表。 |
| GET | `/characters/{id}` | 角色详情。 |
| GET | `/characters/{id}/frames` | 当前角色全部帧数据。 |
| GET | `/characters/{id}/combos` | 当前角色已审核主连招分页和服务端筛选。 |
| GET | `/characters/{id}/combo-filter-options` | 当前角色全部已审核主连招的起手式/标签选项。 |
| GET | `/characters/{id}/combo-parent-options` | 当前角色、指定操作类型下全部已审核顶层连招，供后续压制路线选择父连招。 |
| GET | `/world-tour/combos` | 环球游历已审核主连招分页，按最新上传排序。 |
| GET | `/combos/{id}` | 连招详情；未审核连招只允许作者查看。 |
| GET | `/combos/{id}/followups` | 已审核后续路线。 |
| POST | `/combos` | 登录用户投稿，必须提供演示视频，进入人工审核。 |
| PUT | `/combos/{id}` | 作者修改被驳回的投稿并重新提交。 |
| DELETE | `/combos/{id}` | 作者删除被驳回的投稿。 |
| POST | `/combos/{id}/like` | 点赞/取消点赞已审核连招。 |
| POST | `/combos/{id}/favorite` | 收藏/取消收藏已审核连招。 |

连招分页参数：

- `page`：默认 1。
- `pageSize`：角色详情默认 12。
- `controlType`：`classic` / `modern` / `world-tour`；省略时包含全部操作类型。
- `starter`：规范化为大写后精确匹配。
- `tags`：逗号分隔；多个标签采用 AND 语义。
- `sort`：`damage`（伤害降序）、`likes`（点赞降序）、`createdAt`（上传时间降序，默认）。同值时按 ID 降序保持稳定。

筛选顺序为“数据库全量筛选后分页”，不是只筛当前页。超出最后一页时返回空 `list`，同时保留实际 `total`。筛选选项接口接受可选 `controlType` 参数，返回当前角色对应范围内全部已审核主连招的去重选项：

```json
{
  "starters": ["2MK", "5HP"],
  "tags": ["corner", "counter-hit"]
}
```

父连招选项接口不分页，只返回同角色、同操作类型、已审核通过且自身不是后续路线的顶层连招。写接口会重复执行相同约束，不能通过绕过前端关联待审核、已驳回或另一条后续路线。

连招写入字段：`characterId`、`starter`、`route`、`comboText`、`damage`、`driveCost`、`saCost`、`advantageFrames`、`difficulty`、`cornerOnly`、`controlType`、`routeCharacterIds`、`type`、`tags`、`videoUrl`、`trainingNotes`、`difficultyNote`、`followupParentId`。环球游历连招使用 `routeCharacterIds` 按招式顺序记录角色，数量必须与路线招式数一致，`characterId` 可省略。

## 文件上传

```http
POST /files
Content-Type: multipart/form-data
```

字段：

- `file`：上传文件。
- `usage`：`avatar` 或 `combo-video`。

限制：头像最大 5 MiB；视频最大 100 MiB；校验扩展名、Content-Type 和文件头；默认每位用户总上传配额 1 GiB。未被头像或连招引用且超过保留时间的文件由定时任务清理。

## 通知、举报、反馈和统计

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/notifications` | 当前用户通知分页，支持 `unreadOnly` 和 `type`。 |
| POST | `/notifications/{id}/read` | 标记本人通知已读。 |
| POST | `/notifications/read-all` | 批量标记已读，可按 `type` 限定。 |
| POST | `/reports` | 登录用户举报目标。 |
| POST | `/feedback` | 登录用户提交站点反馈。 |
| GET | `/announcements` | 已发布公告，支持 `limit`。 |
| POST | `/analytics/visit` | 访问埋点；在线请求只追加日志，UV/PV 定时汇总。 |

通知、点赞和收藏以稳定 `user_id` 判断所有权；用户名字段仅作为展示快照。

通知类型仅支持 `combo_review`、`combo_like`、`combo_favorite`、`feedback` 和 `system`。关注动态、评论/回复和公告镜像不属于当前通知范围；不支持的 `type` 会被拒绝。

## 后台管理

后台完整管理员接口：

- 会话与仪表盘：`GET /admin/me`、`GET /admin/dashboard`。
- 用户：`GET /admin/users`、`PUT /admin/users/{id}`、`POST /admin/users/{id}/ban`、`POST /admin/users/{id}/unban`。
- 角色：查询、新增、排序、修改、删除 `/admin/characters`。
- 帧数据：查询、新增、修改、删除、官网同步、同步日志和变更历史 `/admin/frames`。
- 连招：分页查询、新增、修改、审核、重复检测、视频检查/审核、删除 `/admin/combos`。
- 举报：分页查询、单条处理、批量处理 `/admin/reports`。
- 反馈：分页查询、单条处理、批量处理 `/admin/feedbacks`。
- 公告：查询、新增、修改、删除 `/admin/announcements`。
- 通知：`POST /admin/notifications/broadcast`。
- 审计：`GET /admin/audit-logs`。

拥有 `combo_review` 权限的普通角色只可进入连招管理相关后台能力；其他后台能力要求 `role=admin`。
